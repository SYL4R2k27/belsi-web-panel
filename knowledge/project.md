# Project — BELSI.Work

## System Identity

BELSI.Work — платформа управления полевыми рабочими бригадами. Система координирует выполнение работ на объектах, контроль качества через фотоотчёты, управление сменами, задачами, командами и финансовую отчётность.

Это не маркетинговый продукт. Это внутренний инструмент операционного управления.

## Проблема

Координация полевых бригад монтажников требует:
- отслеживания рабочего времени с доказательной базой
- верификации качества работ через фотоотчёты
- управления назначениями и задачами в реальном времени
- прозрачной системы оплаты, привязанной к выполненной работе
- единой точки связи между полевыми сотрудниками и администрацией

Без единой платформы — ручная координация, потеря информации, споры по оплате, отсутствие контроля качества.

## Для кого

### Роли пользователей

| Роль | Описание | Платформа |
|------|----------|-----------|
| `installer` | Полевой монтажник. Выполняет смены, загружает фотоотчёты, получает задачи, общается с куратором | Мобильное приложение |
| `foreman` | Бригадир. Управляет составом команды, создаёт инвайт-коды, мониторит работу бригады | Мобильное приложение |
| `curator` | Администратор-куратор. Полный контроль операций: модерация фото, управление сменами, задачами, финансами, поддержкой | Веб-панель |

### Иерархия

```
curator (1)
  └── foreman (M)
        └── installer (M)
```

Куратор видит всё. Бригадир видит свою команду. Монтажник видит только себя.

## Ключевые функции

### 1. Управление сменами (Shift Tracking)
- Старт / пауза / возобновление / завершение смены
- Вычисление активного, паузного и оплачиваемого времени
- Один активный shift на installer
- Все переходы состояний аудируются

### 2. Фотоотчёты (Hourly Photo Reports)
- Загрузка фото, привязанных к активной смене
- Почасовая индексация (hour_index)
- Модерация куратором: pending → approved / rejected
- При отклонении обязателен комментарий

### 3. Управление задачами (Tasks)
- Создание, назначение, смена статуса, дедлайны
- Статусы: new → in_progress → done / cancelled / overdue
- Полная история назначений и комментариев

### 4. Финансы (Wallet & Payments)
- Кошелёк пользователя с балансом
- Транзакции: shift_payment, manual_adjustment, withdrawal, penalty, bonus
- Выплаты с детализацией по источникам
- Акты за период
- Все финансовые записи неизменяемы после завершения

### 5. Команды и инвайты (Teams & Invites)
- Бригадир создаёт команду и инвайт-коды
- Монтажник активирует инвайт-код для вступления в команду
- Один активный membership на монтажника

### 6. Поддержка (Support Chat)
- Тикеты поддержки с категориями и приоритетами
- Чат-сообщения между монтажником и куратором
- Внутренние заметки куратора (is_internal)
- Все сообщения append-only

### 7. Профиль (Profile Management)
- Личные данные, роль, ставка
- Уникальность телефона (E.164)
- Блокировка/разблокировка куратором

### 8. Инструменты и оборудование (Tools & Equipment)
- Учёт инвентаря: тип, серийный номер, состояние
- Назначение на монтажника
- История выдачи/возврата

## Бизнес-логика: ключевые правила

### Смены
- Максимум одна активная/приостановленная смена на монтажника
- `payable_hours <= actual_hours`
- Переход в `finished`/`cancelled` требует `finished_at`
- Геокоординаты валидируются диапазоном

### Фото
- Уникальность пары `(shift_id, hour_index)` — нет дублей
- `reviewed_by` обязателен при модерации
- `review_comment` обязателен при `rejected`

### Финансы
- Все суммы в `numeric(14,2)`
- Завершённые транзакции неизменяемы
- Корректировки — отдельные записи, не мутация оригинала
- Минимальный срок хранения: 5 лет

### Команды
- Монтажник может быть в одной активной команде
- Куратор не может быть членом команды
- `foreman_id` у installer обязан ссылаться на активного foreman

### Аудит
- Каждое write-действие куратора записывается в audit_log
- Запись включает: актор, действие, сущность, old_data, new_data, IP, timestamp
- Аудит-лог immutable — записи не обновляются и не удаляются

## Метрики успеха

| Метрика | Описание |
|---------|----------|
| Shift completion rate | Процент завершённых смен от начатых |
| Photo moderation turnaround | Время от загрузки фото до модерации |
| Rejected-photo % by team | Процент отклонённых фото по командам |
| Task overdue rate | Процент просроченных задач |
| Support first-response time | Время первого ответа на тикет |
| Payment dispute rate | Процент оспоренных выплат |

## Нефункциональные требования

| Параметр | Целевое значение |
|----------|-----------------|
| Доступность | 99.9% monthly |
| P95 latency (чтение) | < 300 ms |
| P95 latency (запись) | < 600 ms |
| Бэкапы | Ежедневные + PITR |
| Шифрование | TLS in transit, AES-256 at rest |
| Наблюдаемость | Request tracing, метрики, структурированные логи |
| Масштаб | Тысячи одновременных пользователей |
